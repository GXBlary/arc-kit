# Mermaid Wardley-Beta Testing Suite ‚Äî Design Spec

**Date**: 2026-03-28
**Status**: Draft
**Scope**: Standalone test suite validating ArcKit's generated Mermaid `wardley-beta` syntax against the real Mermaid parser from PR #7147

---

## Context

ArcKit's 5 Wardley commands (`wardley`, `wardley.value-chain`, `wardley.climate`, `wardley.doctrine`, `wardley.gameplay`) generate dual output: OnlineWardleyMaps (OWM) syntax and Mermaid `wardley-beta` syntax. The Mermaid format was added in commit 793c9a3.

Mermaid PR #7147 (authored by tractorjuice) merged the `wardley-beta` diagram type to `develop` on 2026-03-17, after the 11.13.0 release. It is not yet in a released version. The pre-release build is available via `npm i https://pkg.pr.new/mermaid@7147`.

We need to validate that the Mermaid syntax our commands generate is accepted by the actual Mermaid parser, and that the maps render correctly to SVG.

## Goals

1. Confirm all currently generated Mermaid wardley-beta syntax parses without errors
2. Confirm SVG rendering produces valid output (no rendering failures)
3. Identify edge cases or syntax quirks that could cause issues in generated maps
4. Explore new Mermaid features not yet used by ArcKit, to determine which are safe to adopt

## Non-Goals

- Integrating validation into the hook system (future work)
- Automated CI pipeline (can be added later)
- Pixel-perfect visual comparison testing

## Test Fixtures

### 01-basic-map.mmd ‚Äî Core Syntax

Tests: `wardley-beta` keyword, `title`, `size [w, h]`, `anchor`, `component` with `[visibility, evolution]` coordinates, `->` dependency links.

Maps to: Core syntax generated by all 5 Wardley commands.

### 02-decorators.mmd ‚Äî Sourcing Strategy Decorators

Tests: `(build)`, `(buy)`, `(outsource)`, `(inertia)`, and combinations like `(buy) (inertia)`.

Maps to: `/arckit.wardley` sourcing strategy output from Build vs Buy analysis.

### 03-pipeline.mmd ‚Äî Pipeline Components

Tests: `pipeline Parent { component "Child" [evolution] }` with multiple children at different evolution positions.

Maps to: Pipeline translation from OWM syntax.

### 04-evolve.mmd ‚Äî Evolution Arrows

Tests: `evolve ComponentName targetEvolution` without `label` text (our commands strip labels since Mermaid doesn't support them on evolve).

Maps to: Evolution movement arrows in generated maps.

### 05-notes-annotations.mmd ‚Äî Notes and Annotations

Tests: `note "text" [vis, evo]`, `annotations [x, y]`, `annotation N,[vis, evo] "text"`.

Maps to: Annotation output from all Wardley commands.

### 06-complex-full.mmd ‚Äî Realistic Full Map

Tests: 20+ components, multiple anchors, pipelines, evolve arrows, decorators, notes, annotations ‚Äî all combined. Representative of what `/arckit.wardley` actually produces for a real project.

Maps to: Full Wardley Map document output.

### 07-edge-cases.mmd ‚Äî Stress Testing

Tests: Component names with spaces (unquoted and quoted), special characters, 30+ components, deep dependency chains, decimal precision (0.01 increments), boundary coordinates (0.00, 1.00).

Maps to: Edge cases that could occur in real generated maps.

### 08-new-features.mmd ‚Äî New Mermaid Features (Exploratory)

Tests features Mermaid supports but ArcKit doesn't yet generate:
- `(market)` decorator
- `evolution Genesis -> Custom Built -> Product -> Commodity` custom stages
- `label [offsetX, offsetY]` position fine-tuning
- `-.->` dashed arrows (weak dependencies)
- `+>` / `+<` / `+<>` flow links
- `+'label text'>` labeled flows
- `A -> B; annotation text` link annotations
- `accelerator` / `deaccelerator`
- `evolution A / B -> C / D` dual labels
- `evolution A@0.25 -> B@0.50` custom stage widths
- `#` comments

Maps to: Feature gap analysis ‚Äî determines which new features are safe to adopt.

## Validation Script (`validate.mjs`)

Imports `mermaid`, iterates over all `.mmd` files in `fixtures/`, runs `mermaid.parse()` on each.

**Behavior**:
1. Read each `.mmd` file from `fixtures/`
2. Call `await mermaid.parse(contents)`
3. Parse succeeds: print `PASS` (green) with fixture name
4. Parse throws: print `FAIL` (red) with fixture name, error message, and line number
5. Print summary: `N/M passed`
6. Exit code: 0 if all pass, 1 if any fail

No test framework dependency ‚Äî just `node validate.mjs`.

## Render Script (`render.mjs`)

Uses `@mermaid-js/mermaid-cli` (mmdc) to render each fixture to SVG.

**Behavior**:
1. Ensure `output/` directory exists
2. For each `.mmd` file in `fixtures/`:
   - Run `mmdc -i <input> -o <output> -p puppeteer-config.json`
   - Success: print `RENDERED` (green) with file size
   - Failure: print `FAILED` (red) with error details
3. Print summary
4. Exit code: 0 if all render, 1 if any fail

**Puppeteer config** (`puppeteer-config.json`):
```json
{
  "args": ["--no-sandbox", "--disable-setuid-sandbox"]
}
```

Required for headless Chrome in the codespace environment.

## Output

- **Parse results**: Console output from `validate.mjs`
- **SVG files**: `output/*.svg` for visual inspection (gitignored)
- **Findings**: Any syntax issues or new feature adoption recommendations documented in a follow-up PR

## Known Risks

1. **Puppeteer in codespace**: `mmdc` requires Chromium. The codespace may need additional dependencies (`libgbm`, `libnss3`, etc.). Fallback: skip SVG rendering if Chromium setup fails; parse validation alone is still valuable.
2. **Pre-release build stability**: The `pkg.pr.new` build is from the merged PR but not a tagged release. Minor API differences possible vs the eventual release.
3. **NAME_WITH_SPACES grammar**: The Mermaid grammar uses a complex regex for unquoted names with spaces. Edge cases in fixture 07 may reveal parsing limits.
